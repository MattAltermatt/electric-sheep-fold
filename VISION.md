# 🔮 Vision

## Why electric-sheep-fold exists

This repo **is** a preserved corpus of [Electric Sheep](https://electricsheep.org)
`.flam3` genomes — a public record of 20+ years of crowdsourced generative
art from the Draves–Reckase distributed renderer project. The corpus is the
deliverable. The `sheep-fold` toolchain in this repo is the means by which
the corpus is built, indexed, and republished.

## The two reasons

1. **Preservation.** The Electric Sheep server (`v3d0.sheepserver.net`) is
   volunteer infrastructure preserving 20+ years of generative art. The
   electricsheep.com archive holds older "dead" generations whose live
   render-servers have long since gone offline. Both are single points of
   failure — `.flam3` genomes are tiny XML files but represent thousands of
   designer-hours; once gone, gone. This corpus is the durable mirror.

2. **Pyr3-facing index.** [pyr3](https://github.com/MattAltermatt/pyr3) is a
   deterministic JVM fractal-flame renderer that claims byte-identical
   output across machines. To prove that claim broadly, pyr3 needs a corpus
   of real flames to render against — not just one golden. The corpus
   `index.json` is queryable by variation, structural features, and
   pyr3-specific limitation flags so an agent can find parity candidates
   in one `jq` filter.

## What "done" looks like

- **Latest GitHub Release** is the canonical corpus snapshot. Consumers
  download `corpus-all.zip` (everything) or per-gen zips (`gen-244.zip` etc.)
  plus `INDEX.md` + `index.json` + `ATTRIBUTION.md`. No git clone required
  to use the data.
- Sticky `missing.txt` per gen records confirmed-empty IDs so future
  preservation cycles never re-probe.
- `corpus/_index/` is regenerated by `sheep-fold index` after any corpus
  mutation; ships with every Release.
- pyr3 consumes the corpus for parity testing (outside this repo's scope
  but the reason the corpus exists in this shape).

## Politeness as a design constraint

The live ES server (v3d0, lighttpd 1.4.33) is volunteer infrastructure.
`sheep-fold` treats it accordingly:

- **20-second default cadence** between requests on v3d0; configurable but
  never parallelized. `LIVE_GENS = {247, 248}` hard-gates which gens the
  live fetcher will hit.
- **2-second cadence** on the electricsheep.com archive (S3-backed; gentler
  on aggregate; modest cross-gen parallelism OK).
- **Sticky 404 memory** so we never re-probe known-empty IDs (the worst
  form of rudeness is also the most pointless — re-asking after we already
  learned the answer).
- **Identifiable User-Agent** that names the project and links the repo,
  so server admins can reach us if our load is ever a problem.

## Storage shape

One sealed `.zip` per generation under `corpus/{gen}/`. The zip contains a
`MANIFEST.csv` as its first entry (id, sha256, fetched_at, source_url,
xform_count, variations, designer nick, …) followed by each `.flam3` file
with its canonical name (`electricsheep.{gen}.{id:05d}.flam3`). The
`MANIFEST.csv` schema is the seam the `index.json` aggregator reads from.

For distribution, the Release-build script renames each per-gen zip to
`gen-{N}.zip` and bundles the lot into `corpus-all.zip`. The Release tag
(e.g. `v0.2.2`) carries the snapshot version; filenames stay stable
across Releases.
