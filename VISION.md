# 🔮 Vision

## Why electric-sheep-fold exists

This repo **is** a preserved corpus of [Electric Sheep](https://electricsheep.org)
`.flam3` genomes — a public record of 20+ years of crowdsourced generative
art from the Draves–Reckase distributed renderer project. The corpus is the
deliverable. The `sheep-fold` toolchain in this repo is the means by which
the corpus is built, indexed, and republished.

## The two reasons

1. **Preservation.** The Electric Sheep server (`v3d0.sheepserver.net`) is
   volunteer infrastructure preserving 20+ years of generative art. The
   electricsheep.com archive holds older "dead" generations whose live
   render-servers have long since gone offline. Both are single points of
   failure — `.flam3` genomes are tiny XML files but represent thousands of
   designer-hours; once gone, gone. This corpus is the durable mirror.

2. **Pyr3-facing index.** [pyr3](https://github.com/MattAltermatt/pyr3) is a
   deterministic JVM fractal-flame renderer that claims byte-identical
   output across machines. To prove that claim broadly, pyr3 needs a corpus
   of real flames to render against — not just one golden. The corpus
   `index.json` is queryable by variation, structural features, and
   pyr3-specific limitation flags so an agent can find parity candidates
   in one `jq` filter.

## What "done" looks like

- **Latest GitHub Release** is the canonical corpus snapshot. Each release
  is tagged by ISO build date (`YYYY-MM-DD`); consumers download
  `corpus-all-{date}.tar.xz` (everything, LZMA2) or per-gen zips
  (`gen-244-{date}.zip` etc.) plus `INDEX.md` + `index.json` +
  `ATTRIBUTION.md`. Per-gen and mega-bundle extract to identical trees
  (overlay invariant). No git clone required to use the data.
- Sticky `missing.txt` per gen records confirmed-empty IDs so future
  preservation cycles never re-probe.
- `corpus/_index/index.json` is regenerated by `sheep-fold index` after
  any corpus mutation; ships with every Release. The v6 envelope carries
  `_schema_version` + `_build_date` + the genome records — five pyr3 AutoRoute
  GPU-safety fields driving CPU-vs-GPU routing, v5 parser-malformation flags,
  plus v6 provenance (`version`) + tone-map (`gamma` / `vibrancy` /
  `estimator_minimum` / `estimator_curve`) fields.
- pyr3 consumes the corpus for parity testing (outside this repo's scope
  but the reason the corpus exists in this shape).

## Politeness as a design constraint

The live ES server (v3d0, lighttpd 1.4.33) is volunteer infrastructure.
`sheep-fold` treats it accordingly:

- **20-second default cadence** between requests on v3d0; configurable but
  never parallelized. `LIVE_GENS = {247, 248}` hard-gates which gens the
  live fetcher will hit.
- **2-second cadence** on the electricsheep.com archive (S3-backed; gentler
  on aggregate; modest cross-gen parallelism OK).
- **Sticky 404 memory** so we never re-probe known-empty IDs (the worst
  form of rudeness is also the most pointless — re-asking after we already
  learned the answer).
- **Identifiable User-Agent** that names the project and links the repo,
  so server admins can reach us if our load is ever a problem.

## Storage shape

The on-disk corpus is **append-only chunked**:
`corpus/{gen}/{bucket}/electricsheep.{gen}.{id:05d}.flam3` where
`bucket = (id // 10000) * 10000` zero-padded. `missing.txt` lives at
`corpus/{gen}/missing.txt` (one per gen, not per bucket). Same shape for
live and dead gens. Mutated only by `fetch-all` / `import` (append a
flam3 OR append an id to `missing.txt`); never re-sealed, never re-keyed.
Bucketing keeps gens browsable in Finder + `ls` even at 30k+ flames each.

For distribution, `sheep-fold release-build [--date YYYY-MM-DD]` reads
the corpus on demand and emits dated release artifacts into
`build/release/`:

- `gen-{N}-{date}.zip` per gen — ZIP DEFLATE-9. Members:
  `MANIFEST.csv` (rebuilt from current state; 11-column schema) +
  `missing.txt` (sticky-404 travels with the artifact) +
  `{bucket}/electricsheep.{N}.{id}.flam3`.
- `corpus-all-{date}.tar.xz` mega-bundle — LZMA preset 6 (~−78% vs zip
  for the full corpus). Members: full corpus tree (`{gen}/MANIFEST.csv`,
  `{gen}/missing.txt`, `{gen}/{bucket}/...`, `_index/`, `ATTRIBUTION.md`).
- `corpus-chunks-{date}.tar` — renderer-facing delivery layer: brotli'd
  256-id chunks + per-gen availability manifests, consumed by pyr3 at
  `pyr3.app/v1/gen/{gen}/id/{id}`. A transfer concern, independent of the
  10k storage bucketing.

The overlay invariant — per-gen zip extracted under `{gen}/` produces
the same on-disk subtree as extracting the mega-bundle and taking the
matching subset — lets consumers grab piecemeal OR all-in-one and the
trees fit together.

Release tag = the same ISO date stamped into the artifact filenames.
The corpus dir is the only mutable state; release artifacts are pure
derivatives, reproducible from corpus + build date.
