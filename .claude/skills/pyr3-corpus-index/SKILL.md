# pyr3-corpus-index

The electric-sheep-fold corpus index is the **agentic query layer** over the
preserved `.flam3` corpus. It's regenerated by walking every sealed zip
(`corpus/{gen}/NNNNN-NNNNN.zip`) and every working-dir chunk, parsing each
flame's XML, and emitting:

- `corpus/_index/index.json` — one JSON record per flame, flat array,
  queryable with `jq` or any JSON tooling.
- `corpus/_index/INDEX.md` — aggregations (per-gen counts, variation
  histogram, structural-feature counts, recipe table) for fast scanning.

This skill is read by agents (humans don't query it directly). Outputs are
not pretty — they're optimized for `jq` / programmatic consumption.

## When to invoke

- User asks for "an example of variation X" / "a flame using Y" → query
  `index.json`.
- User asks "is there a corpus index?" / "show me corpus stats" → cat
  `INDEX.md`.
- User asks "rebuild the index" / "regenerate the corpus index" / "the
  corpus changed, refresh the index" → run `sheep-fold index`.
- After any `import`, `seal`, `unseal`, or chunk-level mutation → re-run
  `sheep-fold index` so the index doesn't drift.

## How to invoke

```sh
# Regenerate (overwrites both outputs, ~45s for the full ~140k-flame corpus)
sheep-fold index

# Custom corpus / output paths
sheep-fold index --corpus /path/to/corpus --out /path/to/out/
```

## Schema (per-flame record in `index.json`)

Every record has:

| Field | Type | Notes |
|-------|------|-------|
| `id` | str | `"{gen}/{sheep_id:05d}"`, e.g. `"244/42746"` |
| `gen` | int | generation number |
| `sheep_id` | int | sheep id within the gen |
| `byte_size` | int | raw `.flam3` size in bytes |
| `kind` | str | `"genome"` \| `"animation"` \| `"corrupt"` |
| `valid` | bool | parse result |
| `sealed` | bool | true if from a sealed `.zip`; false if from a working dir |

**Genomes (`kind == "genome"`)** add:

| Field | Type | Notes |
|-------|------|-------|
| `frame_count` | int | always `1` for genomes |
| `name`, `nick`, `url` | str | designer attribution (`nick` present = human-designed) |
| `dims` | str | e.g. `"1280 720"` |
| `rotate`, `brightness` | float | flame-level scalars |
| `palette_mode` | str | `"step"` or `"linear"` |
| `filter_shape` | str | `"gaussian"` is the default |
| `background` | [r, g, b] | flame background color |
| `supersample` | int | flam3 supersample (default 1); **pyr3 needs 1 for parity** |
| `highlight_power` | float | default -1; **pyr3 NotImplementedError if ≥0** |
| `has_symmetry` | bool | flame-level `<symmetry>` child present |
| `symmetry_kind` | int | only if `has_symmetry` (palette symmetry kind) |
| `xform_count` | int | number of `<xform>` siblings |
| `has_final_xform` | bool | `<finalxform>` child present |
| `has_post_affine` | bool | any xform has non-identity `post` |
| `has_chaos` | bool | any xform has `chaos` (xaos) attribute; **pyr3 doesn't model xaos** |
| `negative_weight_xforms` | int | count of xforms with `weight < 0` |
| `variations` | [str] | sorted unique variation names across all xforms (incl. finalxform) |

**Animations (`kind == "animation"`)** add:

| Field | Type | Notes |
|-------|------|-------|
| `frame_count` | int | number of `<flame>` siblings (the keyframe count) |

Animations are multi-flame morphs / interpolation snapshots — derivative
content. **pyr3 doesn't render them directly.** Agentic queries should
default to `kind == "genome"`.

**Corrupt (`kind == "corrupt"`)** add:

| Field | Type | Notes |
|-------|------|-------|
| `error` | str | short reason (e.g. `"zero-byte"`, `"parse: junk after document element"`) |

## Recipes — `jq` queries an agent will reach for

All recipes assume `cwd` is the repo root and the index lives at
`corpus/_index/index.json`.

**Find flames using a specific variation:**
```sh
jq -r '.[] | select(.kind == "genome" and (.variations | index("bipolar"))) | .id' corpus/_index/index.json | head
```

**Find pyr3-parity-friendly genomes** (no chaos, no supersample override
needed, no HSV-desat NotImplementedError trigger, has finalxform):
```sh
jq -r '.[] |
  select(.kind == "genome"
    and (.has_chaos | not)
    and .supersample == 1
    and .highlight_power < 0
    and .has_final_xform)
  | .id' corpus/_index/index.json | head
```

**Find rare-variation candidates** (good for stress tests — `INDEX.md` lists
counts; pick a variation with single-digit usage):
```sh
jq -r '.[] | select(.kind == "genome" and (.variations | index("separation"))) | .id' corpus/_index/index.json
```

**Find low-complexity baseline genomes:**
```sh
jq -r '.[] | select(.kind == "genome" and .xform_count <= 2 and (.has_chaos | not)) | .id' corpus/_index/index.json | head
```

**Find human-designed genomes** (has nick → CC BY-licensed):
```sh
jq -r '.[] | select(.kind == "genome" and .nick != "") | "\(.id)\t\(.nick)"' corpus/_index/index.json | head
```

**Inspect one flame in full:**
```sh
jq '.[] | select(.id == "244/42746")' corpus/_index/index.json
```

**Find a genome with a specific variation combo:**
```sh
jq -r '.[] |
  select(.kind == "genome"
    and (.variations | index("disc"))
    and (.variations | index("bubble"))
    and (.has_chaos | not))
  | .id' corpus/_index/index.json | head
```

## Where to look up a flame's actual bytes

The index gives you the `id` (e.g. `"244/42746"`). To extract the raw
`.flam3` content:

```sh
# Find which sealed zip contains it (gens 165-245 = whole-gen zip; 247/248 = chunked)
python3 -c "
import zipfile
from pathlib import Path
gen, sid = '244', '42746'
for zp in sorted(Path(f'corpus/{gen}').glob('?????-?????.zip')):
    with zipfile.ZipFile(zp) as zf:
        name = f'electricsheep.{gen}.{sid:>05}.flam3'
        if name in zf.namelist():
            print(zf.read(name).decode())
            break
"
```

## Maintenance

- After **any** corpus mutation (`import`, `seal`, `unseal`, `fetch-all`)
  re-run `sheep-fold index` so `index.json` doesn't drift from reality.
- The index is fully derivable from the corpus; safe to delete and rebuild.
- ~45s for the full corpus (~140k flames). Stdlib-only; no extra deps.
- Output goes to `corpus/_index/` by default; gitignored along with the
  rest of `corpus/`.

## 📋 Pending hand-off — per-xform variation-count fields

**Status:** queued (recorded 2026-05-22). Triggered by **pyr3 v0.16** surfacing
the GPU `MAX_VARIATIONS_PER_XFORM = 4` cap — current `variations[]` is the
genome-level UNION, which loses per-xform density. The pyr3 session is paused
on this question; once the index is rebuilt with these fields and pushed to
`origin/main`, pyr3 pulls and queries directly to make a data-driven decision
on whether v0.17 should bump the cap to 8 or 16.

### Schema additions (per-genome entry)

Add these fields alongside the existing `xform_count` / `variations[]` /
`has_symmetry` / etc:

| Field | Type | Notes |
|-------|------|-------|
| `xform_var_counts` | `[int]` | per-xform array, `len == xform_count` (pre-symmetry-expansion); preserves full distribution |
| `max_var_per_xform` | int | `max(xform_var_counts)` |
| `mean_var_per_xform` | float | `mean(xform_var_counts)` rounded to 2 decimals |
| `xforms_with_5plus_vars` | int | count of xforms exceeding pyr3:gpu's current cap of 4 (auto-route gating signal) |
| `final_xform_var_count` | int? | variations on `<finalxform>` if present, else `null` |

The full per-xform array is most flexible — downstream tools compute their own
aggregates without re-scanning `.flam3` files. The 3 derived fields are cheap
convenience.

### Counting logic (verbatim flam3 semantics)

A "variation" on an `<xform>` is any attribute whose **name matches the
canonical flam3 variation name set** (the same names already populated in the
current `variations[]` field — the `var_t` enum, 101 entries including the
two ES-corpus extras `hemisphere` + `post_curl`). The attribute VALUE is the
variation's weight (per-variation), distinct from the xform pick-weight
(`weight=` attribute).

```xml
<xform weight="0.5" color="0" coefs="..." linear="0.3" spherical="0.7"/>
<!-- ⇒ 2 variations (linear, spherical) -->

<xform weight="0.33" coefs="..." pdj="0.41" pdj_a="..." pdj_b="..."
       pdj_c="..." pdj_d="..." blade="0.17"/>
<!-- ⇒ 2 variations (pdj, blade). pdj_a/b/c/d are PARAMS of pdj, not
     variations themselves. Filter by EXACT name match against var_t;
     do NOT count params (they all have underscore-suffix names). -->
```

**Edge cases:**
- **`<finalxform>` elements** — include in `xform_var_counts` (treat as a
  regular xform); also surface `final_xform_var_count` as a sibling for
  downstream tools that want to isolate it.
- **`oscope` ↔ `oscilloscope` alias** — both map to the same variation. Count
  once per xform if both appear (unusual; defensive).
- **Purely-affine xforms** (zero variations) — count `0`, include in array.

### Validation

After regenerating:
- `len(xform_var_counts) == xform_count` for every entry
- `sum(xform_var_counts) >= len(variations[])` (sum counts each occurrence;
  `variations[]` dedupes)
- `max_var_per_xform == max(xform_var_counts)`
- `mean_var_per_xform == round(mean(xform_var_counts), 2)`
- `xforms_with_5plus_vars == sum(1 for n in xform_var_counts if n >= 5)`

Hand-verify against a known sample (e.g. `244/00000` — count variations per
xform manually, compare).

## Why these specific fields

The pyr3-limitation fields (`has_chaos`, `supersample`, `highlight_power`)
are not generic — they're chosen because pyr3 has **known limitations** in
those areas (see `../pyr3/parity/goldens/*/README.md` for the running list).
An agentic query for "a flame pyr3 can parity-render" needs to filter these
out. If pyr3 grows new limitations or graduates old ones, extend
`src/electric_sheep_fold/index.py` and document the new field here.

The variation set is the canonical flam3 `var_t` enum (101 entries including
two ES-corpus extras: `hemisphere`, `post_curl`). Any xform attribute NOT
in this set is either a scalar field (weight, color, opacity, …) or a
variation parameter (`julian_power`, `blob_low`, …) — those are ignored
for the variation-usage count.
