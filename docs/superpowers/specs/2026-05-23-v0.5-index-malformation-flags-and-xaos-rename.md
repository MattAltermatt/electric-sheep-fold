# v0.5 — index malformation flags + xaos rename (schema v5)

**Status:** approved · **Date:** 2026-05-23 ·
**Builds on:** [v0.4 chunked + dated + index v4](2026-05-23-v0.4-chunked-dated-release-and-index.md)

## Why this exists

Inbound design proposal from pyr3 (sibling repo, v0.32 in flight) flagged
~153 of 149,904 corpus genomes (≈0.10%; gen-247 worst at ≈0.77%) carry
`<flame center="nan nan" scale="nan">` — textually invalid floats that no
flam3-lineage renderer can interpret. Every downstream consumer (pyr3
today, anything that follows) reinvents a luminance-based discard pass to
skip these. The structural fact belongs in the corpus index, not in every
consumer's render loop.

While the schema is being touched, two adjacent cleanups land in the same
bump:

1. **`symmetry_kind` becomes schema-honest.** v0.4 emits the field only
   when `<symmetry>` is present; consumers must guard `?? 0` or
   `// null` at every read. v0.5 always emits the field — `int` when
   present, `null` when absent — so jq selects like
   `select(.symmetry_kind == 5)` and `select(.symmetry_kind != null)`
   both work without per-record guard.
2. **`has_chaos` → `has_xaos` rename.** The XML attribute is `chaos`,
   but the community-standard feature name (flam3, Apophysis, JWildfire,
   pyr3) is `xaos`. The v0.4 INDEX.md already aliased the heading as
   `has_chaos (xaos)`; this collapses the dual-naming into the
   canonical form.

## Scope

Index-side only. No changes to disk layout, release artifact format,
fetch/import behavior, MANIFEST.csv, or the overlay invariant. The v0.4
artifact wiring carries forward unchanged; only the per-record contents
of `_index/index.json` change shape.

## Invariant changes (v0.4 → v0.5)

### `index.json` envelope

`_schema_version` bumps from 4 to 5. Consumers MUST check the version
before applying v5 jq paths. Forward-compatibility is NOT promised — a
genome record in v5 is structurally different from v4 (new fields,
changed `symmetry_kind` semantics, renamed `has_chaos` → `has_xaos`).

```json
{
  "_schema_version": 5,
  "_build_date": "YYYY-MM-DD",
  "genomes": [...]
}
```

### New per-genome fields

- **`has_nan_camera: bool`** — `True` iff the `<flame>` root element
  has a `center` or `scale` attribute whose value contains a `nan`
  token (case-insensitive, whole-word). Matches the canonical
  malformation pyr3 named (`center="nan nan" scale="nan"`).
- **`has_nan_in_xforms: bool`** — `True` iff any `<xform>` or
  `<finalxform>` element has any attribute value containing a `nan`
  token. Covers affine coefs (`coefs`, `post`) and any variation
  parameter that landed as NaN.

Both fields apply only to `kind == "genome"` records. Animations + corrupt
records do not emit them (consistent with how the existing structural
fields work — animations get `frame_count` only).

### Changed per-genome field

- **`symmetry_kind: int | null`** — was `int` only when `<symmetry>`
  was present in v0.4. v5 always emits the field. `null` when no
  `<symmetry>` element; `int` value of the `kind` attribute when
  present. The companion `has_symmetry: bool` is unchanged and stays —
  it's the cheap predicate; `symmetry_kind` is the stratifier.

### Renamed per-genome field

- **`has_xaos: bool`** (was `has_chaos`) — same value semantics: `True`
  iff any `<xform>` or `<finalxform>` element has a `chaos` attribute.
  The XML attribute name `chaos` is unchanged in the source; only the
  index field name changes.

### Unchanged fields

`supersample`, `has_density_estimator`, `has_hyper_trig`, `has_edisc`,
`max_abs_affine_coef`, `xform_count_post_symmetry`, and every other v0.4
field carry forward identically. The five AutoRoute GPU-safety fields
shipped in v0.4 are not touched.

## Detection rules (load-bearing)

### NaN token detection

A "nan token" is the substring `nan` (case-insensitive), bounded by
non-alphanumeric characters or string boundaries. Implementation: regex
`(?i)(?<![a-z0-9])nan(?![a-z0-9])` matched against attribute string
values.

- ✅ Matches: `"nan nan"`, `"NaN"`, `"1 nan 0 1 0 0"`, `"-NAN"`.
- ❌ Does NOT match: `"banana"`, `"unanchored"`, `"nano"`, `"snanapse"`.

`inf` / `-inf` / `infinity` are NOT in scope for v0.5. Inf has a
different rendering failure mode (dim output vs all-black) and pyr3 has
not named a use case. If wanted later, ships as `has_nonfinite_*`
fields in a future schema bump.

### Scope of attributes scanned

- **For `has_nan_camera`:** only `center` and `scale` attributes on the
  root `<flame>` element. Other root attrs (`name`, `nick`, `url`,
  `size`, etc.) are not numeric and not scanned.
- **For `has_nan_in_xforms`:** ALL attributes of every `<xform>` and
  `<finalxform>` element. This covers `coefs`, `post`, `weight`,
  `color`, `chaos`, and every variation parameter. The whole-word
  regex keeps false positives at zero (no variation name contains the
  isolated token `nan`).

### Why XML-attr iteration, not raw-bytes scan

The malformed genomes parse successfully through `ET.fromstring()` —
`center="nan nan"` is well-formed XML, just numerically invalid. We
already have parsed `Element` objects in `_index_genome()`; iterating
their `.attrib` dicts is a few lines and stays in the existing parse
pass.

## Test plan

`tests/test_index.py` gets:

1. **NaN-camera fixture:** flam3 with `<flame center="nan nan"
   scale="nan">` → `has_nan_camera=True`, `has_nan_in_xforms=False`.
2. **NaN-in-xform fixture:** flam3 with valid camera but
   `<xform coefs="1 nan 0 1 0 0" ...>` → `has_nan_camera=False`,
   `has_nan_in_xforms=True`.
3. **NaN-in-finalxform fixture:** valid camera + valid xforms +
   `<finalxform post="nan 0 0 1 0 0" ...>` → `has_nan_in_xforms=True`.
4. **Case-insensitive variant:** `coefs="1 NaN 0 1 0 0"` → `True`.
5. **False-positive guard:** attribute value containing `"banana"` /
   `"unanchored"` → both NaN flags `False`.
6. **Clean fixture:** existing test_basic_genome should assert BOTH
   new fields are `False`.
7. **`symmetry_kind=null` fixture:** genome with no `<symmetry>`
   element → `symmetry_kind is None` (the existing
   test_genome_with_chaos already covers this implicitly; assertion
   tightens).
8. **`symmetry_kind=int` fixture:** existing
   test_genome_with_symmetry covers this — kept.
9. **`has_xaos` field name:** all four existing `has_chaos` asserts
   renamed.
10. **Schema version:** new assertion that `build_index()` writes
    `_schema_version == 5`.

Tests stay pure / mock-driven; no real-corpus reads.

## Verification path

After impl ships:

```bash
sheep-fold index    # regen against local corpus
jq '[.genomes[] | select(.has_nan_camera)] | length' \
    corpus/_index/index.json
# expected: ~153 (pyr3-reported figure; ±5% acceptable)

jq '[.genomes[] | select(.has_nan_camera) | .gen] |
    group_by(.) | map({gen: .[0], count: length})' \
    corpus/_index/index.json
# expected: gen-247 should be largest bucket

jq '.genomes[0]._schema_version, ._schema_version' corpus/_index/index.json
# expected: 5
```

Pyr3 adopts `supersample` as-is (name already matched the proposal),
renames any internal reference from `has_chaos` to `has_xaos`, and
starts consuming `has_nan_camera`, `has_nan_in_xforms`, and the
always-present `symmetry_kind` on its next index-pull cycle.

## Non-goals

- **Renderer-specific quality flags.** "Renders as single dot," "has
  visible GPU banding," "dim relative to flam3" — consumer-specific
  observations, not structural facts. Each consumer keeps its own
  divergence log.
- **Inf detection.** Out of scope; different failure mode, no named
  consumer.
- **Backward-compat schema v4 reader.** Consumers check
  `_schema_version` and re-pull. Same policy as v3→v4.
- **Re-cutting the v0.4 / 2026-05-23 release tag.** v0.5 ships in the
  next dated release. v0.4 stays as it landed.

## Release model

Ships in the next dated release (tag `YYYY-MM-DD` where YYYY-MM-DD is
the day `release-build` runs). Tag title:
`"Corpus snapshot YYYY-MM-DD — index schema v5"`. CHANGELOG entry
documents the rename + new fields + schema bump.
