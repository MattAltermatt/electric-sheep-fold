# 🐑 electric-sheep-fold v0.3 — loose corpus, built release artifact

**Date:** 2026-05-22 · **Status:** draft → user-approved 2026-05-22 ·
**Supersedes:** [v0.2 chunked-zip](2026-05-20-electric-sheep-fold-v0.2-chunked-zip.md),
[v0.2.1 dead-gen whole-zip](2026-05-21-electric-sheep-fold-v0.2.1-dead-gen-whole-zip.md),
v0.2.2 (whole-gen unification, no spec doc).

The "sealed-immutable whole-gen zip" model conflates the on-disk corpus
state with the distribution artifact. v0.3 separates them: corpus is
loose `.flam3` files + `missing.txt`; release zips are built on demand.
The "chunk" and "seal" concepts retire from the working path entirely.

---

## 🎯 1. Motivation

v0.2.2 collapsed all gens (live + dead) into single whole-gen sealed
zips. Sealed-immutable was clean for the 8 dead gens (truly closed) but
hostile to live gens (247, 248) which are append-only by nature. Each
new `fetch-all` snapshot demanded a reseal, and reseal lost data:

- **The 247/248 sticky-404 incident.** `missing.txt` lives at
  `corpus/{gen}/missing.txt` — **outside** the sealed zip. When the
  v0.2.2 chunk-shape collapse rebuilt the working dir, the prior
  `missing.txt` was overwritten with whatever the in-memory `MissingSet`
  happened to hold. Sticky-404 provenance for those gens was destroyed.
  BACKLOG entry "Embed MISSING.csv in sealed zips" captured the symptom;
  v0.3 fixes the root cause.
- **Fight every snapshot.** "Sealed-immutable + range-completion seal
  trigger" means each new live-gen snapshot needs an unseal-reseal
  round-trip. The "chunk = decade of ids" abstraction earned its keep for
  10k-chunked live gens before v0.2.2; once collapsed to whole-gen, it
  earns nothing. The seal is bureaucracy on a moving target.

The architectural mistake was using one data shape for two jobs. v0.3
gives each job its own shape.

## 📐 2. The split

| Concern | v0.2.x | v0.3 |
|---|---|---|
| On-disk corpus | sealed `NNNNN-NNNNN.zip` + `missing.txt` | loose `.flam3` files + `missing.txt` |
| Distribution artifact | the sealed zip itself | `build/release/gen-{N}.zip` built on demand |
| Sticky-404 store | loose `missing.txt`; not in zip | loose `missing.txt`; **also inside** release zip |
| Live-gen append | reseal whole zip | write one file to flat dir |
| Dead-gen state | frozen sealed zip | frozen flat dir (functionally identical) |
| Index regen | walks zips + loose | walks loose only (faster, no zip seek) |

The on-disk shape is identical across live and dead gens — the gen's
biography is no longer encoded in the data shape. (For history, see the
v0.2.1 spec's "biography in chunk shape" rationale, now retired.)

## 🗂️ 3. Layout

```
corpus/                                  # gitignored, on-disk corpus
├── {gen}/
│   ├── electricsheep.{gen}.{id}.flam3   # loose files, flat dir
│   ├── electricsheep.{gen}.{id}.flam3   # filename preservation invariant
│   ├── ...
│   └── missing.txt                      # sticky-404 store (id-per-line)
├── _index/                              # regenerated by `sheep-fold index`
│   ├── index.json
│   └── INDEX.md
├── _unseal-verified.json                # post-migration id-count audit
└── _live-fetch-logs/                    # daemon logs (unchanged)

build/release/                           # gitignored, built on demand
├── gen-{N}.zip                          # per-gen release; MANIFEST.csv + missing.txt + flat .flam3s
├── corpus-all.zip                       # mega-bundle for full-corpus download
├── INDEX.md
├── index.json
└── ATTRIBUTION.md

build/v0.2-snapshot/                     # gitignored, pre-migration fallback
└── gen-{N}.zip                          # v0.2.5 sealed-shape artifacts
```

Flat dirs of up to ~86k files (gen 244 worst case) — confirmed safe on
APFS. No sharding.

## 🔄 4. Lifecycle

### 4.1 Live fetch (gens 247, 248)

```
sheep-fold fetch-all --gen 247 --upper 50000
  → for id in [0, upper):
      if file exists at corpus/247/electricsheep.247.{id}.flam3: skip-local
      if id in missing.txt: skip-known-missing
      else: HTTP GET; on 200 → atomic write to flat path; on 404 → append to missing.txt
```

No chunks. No seal. The flat dir IS the working state IS the canonical
state. Each `fetch-all` invocation extends the corpus by whatever new ids
appeared upstream.

### 4.2 Dead-gen import (gens 165, 169, 191, 198, 242–245)

After the v0.3 migration runs once, all 8 dead gens are already flat
dirs. New dead-gen preservation (hypothetically, e.g. if ES rolls a new
dead gen) follows the v0.2.1 flow but without the seal step:

```
scripts/scrape_archive_gen.py --gen N         # populates corpus/_scrape-N/
sheep-fold import corpus/_scrape-N/ --gen N   # whole-gen flag retires
  → copy .flam3 files into corpus/N/
  → merge _missing_404.txt into corpus/N/missing.txt
```

`--whole-gen` becomes the default and only mode; the flag is removed.

### 4.3 Release build

```
sheep-fold release-build --corpus corpus/ --out build/release/
  → for each gen subdir:
      build MANIFEST.csv from .flam3 metadata extraction
      pack MANIFEST.csv + missing.txt + .flam3 files → build/release/gen-{N}.zip
  → regenerate index → copy index.json + INDEX.md + ATTRIBUTION.md to out
  → build corpus-all.zip mega-bundle

scripts/build_release.sh                       # thin wrapper around above
```

Release zips are pure derivatives. Deterministic from corpus state
(modulo timestamps). Never live in `corpus/`.

## 🚚 6. Migration

One-time `sheep-fold unseal` per gen, 6-step SIGKILL-safe state machine:

```
(a) snapshot zip       → build/v0.2-snapshot/gen-{N}.zip (idempotent)
(b) extract zip        → corpus/{gen}/.unseal-tmp/
(c) verify             → file count + sha256 vs MANIFEST.csv
(d) atomic-move files  → os.replace each .flam3 into corpus/{gen}/
(e) audit MANIFEST     → os.replace MANIFEST.csv → corpus/{gen}/MANIFEST.csv
(f) commit             → delete source zip + .unseal-tmp/; append row to corpus/_unseal-verified.json
```

State marker `corpus/{gen}/.unseal-state` (`extracted` / `verified` /
`committed`) makes the process resumable after a crash.

Post-migration, `_unseal-verified.json` records per-gen `loose_count`
and `missing_count`. The fetch daemon at startup calls
`verify_unseal_consistency` — if any gen's current
`len(glob(*.flam3)) + len(missing)` diverges from verified counts, the
daemon refuses to start with a "run unseal first" message. This guards
against silent re-fetch of already-known ids.

## 🔒 7. Invariants

**Preserved (unchanged):**

- **Politeness:** 20s ±5s jitter live, 2s ±1s archive.
- **Atomic writes:** `.tmp` + `os.replace` for every flam3 and missing.txt save.
- **Filename preservation:** `electricsheep.{gen}.{id}.flam3` — never rename, strip, re-encode.
- **Sticky-404 never re-probed.** Once in `missing.txt`, never re-fetched.
- **Live-gen scope hard-restricted** to `{247, 248}` in `layout.py`.
- **MPG-only gens** (`old`, `very-old`, `23`, `202`) permanently out of scope.
- **MANIFEST.csv schema:** 11 columns, unchanged. Now generated by `release-build`, not `seal`.

**Retired:**

- **Sealed-immutable.** No more sealed zips on the canonical path. Release zips are derivatives; they can be rebuilt at will.
- **Range-completion is the seal trigger.** No seal trigger; the corpus is the corpus, full stop.
- **Chunk shape: whole-gen for every gen.** No chunks. The flat dir IS the unit of organization.

**New:**

- **Loose-corpus, append-only.** `corpus/{gen}/` is a flat dir of `.flam3` files + `missing.txt`. Mutated only by `fetch-all` / `import`. Never deleted (other than the one-time `unseal` consuming the v0.2 zip).
- **Release-built on demand.** Release zips live in `build/release/`, never `corpus/`. Pure derivative; reproducible from corpus state.
- **Daemon-verified id counts post-migration.** `unseal` writes `corpus/_unseal-verified.json`; daemon refuses to start if counts diverge.

## 🌐 8. Release artifact compatibility

Consumer shape is **identical** to v0.2.2:

- v0.2.2 `gen-{N}.zip` contains: `MANIFEST.csv` + flat `electricsheep.{N}.*.flam3` files.
- v0.3 `gen-{N}.zip` contains: `MANIFEST.csv` + flat `electricsheep.{N}.*.flam3` files + **`missing.txt`** (new).

v0.3 release is a strict superset; downstream consumers (pyr3, agentic
queries) get the new `missing.txt` "for free" and can ignore it if they
don't care. No compat layer needed. `corpus-all.zip` mega-bundle stays
nested-zips per the v0.2.2 README convention.

## 🚦 9. Daemon resume point + gap-id catchup

When the live-fetch daemon was stopped at the start of Phase A:

- **PID:** 61017 (orphaned to init via nohup+disown; no chain wrapper still alive)
- **Last fetched id:** gen 247, id 32084 (200 OK, downloaded)
- **Last recorded missing:** gen 247, id 32085 (404)
- **In-flight `.tmp` files at kill time:** 0 (clean stop)
- **Log:** `corpus/_live-fetch-logs/fetch-live-20260522-0705.log`

Phase G resumes the daemon on the v0.3 code path. Because skip-local is
satisfied by `flam3_path(...).exists()` and skip-known-missing by
`missing.txt`, the daemon picks up at the next unknown id.

**Gap-id catchup (intentional, not a bug).** v0.2.4 used the sealed zip's
filename range as a skip-set ("range-trust"): once `corpus/247/00000-29999.zip`
existed, ids in `[0, 30000)` were treated as decided even when not present
in the namelist OR `missing.txt`. v0.3 retires range-trust; `missing.txt`
is the sole source of truth. Gens 247 + 248 had ~21k + ~11k gap-ids
respectively (in-range, not on disk, not in `missing.txt`) that v0.2.4
silently skipped. v0.3 treats these as unknown, so the daemon politely
re-probes them at the 20s cadence — about 7.4 days of background work,
non-blocking. Each result (404 → append to `missing.txt`; 200 → atomic
write to flat path) is durably recorded; SIGKILL-safe; resumable.

**Why not bulk-backfill `missing.txt` instead?** Trusting v0.2.4's
range-trust assumption would mark ids as sticky-404 forever. Per the
sticky-404 invariant ("never re-probed"), any false-positive 404 would
silently lose a real 200 upstream. The 7.4-day catchup is the cost of
ground-truth state. See v0.2.4 CHANGELOG for the historical context that
made range-trust necessary.

## 🚫 11. Out of scope

- Compression beyond zip DEFLATE level 9 (already optimal for this content).
- Sharded directory layout (`{id_div_1000}/...`). APFS handles 86k flat fine.
- Re-introducing the `seal` concept under any other name.
- Backports of the v0.3 layout to v0.2.x releases. v0.2.5 is the final v0.2 release.
