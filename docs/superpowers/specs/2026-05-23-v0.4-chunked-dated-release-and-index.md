## Context

Two months into v0.3 (loose-corpus, append-only), three pressures
converge into one deliberate invariant evolution:

1. **Filesystem + human ergonomics.** Big gens — gen-244 at 33,594 files,
   gen-198 at 31,836 — live in a single flat directory. APFS and ext4
   don't care, but Finder lag is real, `ls` produces walls of output,
   and human pattern-recognition of "what id ranges do we have?" is
   hostile.
2. **Mega-bundle compression leaves money on the table.** Current
   `corpus-all.zip` is 499 MB across 166,600 flames; re-packed as
   `corpus-all.tar.xz` it measures ~110 MB (−78%) because LZMA builds a
   cross-file dictionary across redundant flame XML, which per-file zip
   headers prevent. The v0.2.5 deferral cited "loses random-access into
   the archive," but under v0.3 the consumer pattern (pyr3, agentic
   queries, humans) is download-and-extract-whole, never seek-within.
   The random-access objection is no longer load-bearing.
3. **Pyr3 keeps reparsing flame XML to pick CPU vs GPU.** Pyr3's
   `AutoRoute.kt:73 verdict(genome)` queries five attributes
   `index.json` doesn't currently expose. Pyr3 BACKLOG carries a literal
   "ESF `has_nan_camera` field" entry naming the gap.

Gens 247 + 248 just finished downloading on 2026-05-23.

## Invariant changes (v0.3 → v0.4)

### Disk layout

**v0.3:**
> Loose-corpus, append-only. `corpus/{gen}/` is a flat dir of
> `electricsheep.{gen}.{id}.flam3` files + a `missing.txt`.

**v0.4:**
> Chunked loose-corpus, append-only.
> `corpus/{gen}/{bucket}/electricsheep.{gen}.{id}.flam3` where
> `bucket = f"{(id // 10000) * 10000:05d}"`. Per-gen `missing.txt`
> stays at `corpus/{gen}/missing.txt`.

The filename-preservation invariant (`electricsheep.GGG.NNNNN.flam3`)
is unchanged — only the parent directory gains a layer. The chunking
rule applies uniformly to every gen, even small ones; gen-165 (998
files) lives entirely in `corpus/165/00000/`. Uniform layout, no
threshold rule, no special-case code.

### Release artifact naming

**v0.3:** `gen-NNN.zip` + `corpus-all.zip`
**v0.4:** `gen-NNN-YYYY-MM-DD.zip` + `corpus-all-YYYY-MM-DD.tar.xz`

`YYYY-MM-DD` is the build date (the day `sheep-fold release-build`
ran). One date shared across every artifact in a release. ISO-8601
format so artifact filenames sort chronologically.

### Release artifact format

**v0.3:** ZIP DEFLATE-9 for both per-gen artifacts and the mega-bundle.
**v0.4:** ZIP DEFLATE-9 for per-gen; `tar.xz` for the mega-bundle.

Rationale: the per-gen winner depends on file count. Small gens
compress better in ZIP (gen-248 is 13 MB as ZIP vs 19 MB as tar.xz —
DEFLATE-9 already exploits within-file XML repetition and xz overhead
drowns the cross-file dictionary win). Large gens compress much better
in tar.xz (gen-244 is 204 MB as ZIP vs 45 MB as tar.xz). The
mega-bundle aggregates across all gens and consistently wins under
tar.xz (~−78%). The optimal shape is therefore the hybrid below — not
a single format applied uniformly.

## 🪨 Load-bearing structural invariant — the overlay property

**Per-gen zips and the mega-bundle produce the same on-disk tree.**
A consumer can grab the artifacts piecemeal OR all in one — they will
fit together regardless. This is the central UX promise of v0.4 and
must be surfaced prominently in README + the pyr3-corpus-index skill
so external consumers know they have the choice.

```sh
# Path A — bulk consumer, one artifact
mkdir corpus && cd corpus
tar -xJf ../corpus-all-2026-05-23.tar.xz
# Result: corpus/{165, 169, 191, 198, 242, 243, 244, 245, 247, 248}/
#         each with missing.txt + bucket dirs + MANIFEST.csv
#         plus _index/{index.json, INDEX.md} and ATTRIBUTION.md
```

```sh
# Path B — piecemeal, only the gens the user wants
mkdir corpus && cd corpus
unzip ../gen-244-2026-05-23.zip -d 244/
unzip ../gen-247-2026-05-23.zip -d 247/
# Result: corpus/244/, corpus/247/, identical content to Path A's subset
```

```sh
# Path C — combine: bulk first, then patch a single gen from a later release
tar -xJf ../corpus-all-2026-05-23.tar.xz
unzip -o ../gen-244-2026-06-01.zip -d 244/
# Result: corpus/ tree with 244 patched, rest unchanged
```

All three paths leave the same shape on disk:

```
corpus/{gen}/missing.txt
corpus/{gen}/MANIFEST.csv
corpus/{gen}/{bucket}/electricsheep.{gen}.{id}.flam3
corpus/_index/{index.json, INDEX.md}
corpus/ATTRIBUTION.md
```

Verifiable: `diff -r overlay-via-pergen overlay-via-bundle` is empty
after extracting both into staging directories. The overlay test is
part of the release-build verification suite
(`tests/test_release.py::test_overlay_invariant`).

## Architecture

### 1. Chunked corpus layout — single source of truth

A single helper in `layout.py` is the only place that converts
`(gen, sheep_id) → disk path`. Every fetch / import / release / index
call-site routes through it:

```python
def bucket_for(sheep_id: int) -> str:
    return f"{(sheep_id // 10000) * 10000:05d}"

def flam3_path(corpus_root: Path, gen: int, sheep_id: int) -> Path:
    return (
        corpus_root
        / str(gen)
        / bucket_for(sheep_id)
        / f"electricsheep.{gen}.{sheep_id:05d}.flam3"
    )
```

Bucket dirs are created lazily by `os.makedirs(..., exist_ok=True)`
at write time. The chunked layout adds at most one bucket-dir-create
per new id, negligible overhead.

### 2. One-shot migration (`sheep-fold migrate-chunked`)

A new CLI subcommand reshapes an existing flat v0.3 corpus into v0.4
chunked form:

- For each gen dir, scan `corpus/{gen}/electricsheep.{gen}.*.flam3`.
  For each file, compute its bucket and `os.replace` it into
  `corpus/{gen}/{bucket}/`. Atomic at the filesystem level;
  SIGKILL-safe (no half-moved file).
- `missing.txt` stays at `corpus/{gen}/missing.txt`.
- Idempotent: a file already inside `corpus/{gen}/{bucket}/` is a no-op.
- Writes `corpus/_chunked-verified.json` recording per-gen
  `{loose_count, missing_count, bucket_count}`. Replaces the v0.3
  `_unseal-verified.json`.

`fetch-all` reads `_chunked-verified.json` at startup and refuses to
start if any gen still has files at the flat path, with a "run
migrate-chunked first" message. Mirrors the v0.3 unseal guard.

### 3. Release build (`sheep-fold release-build`)

Emits to `build/release/`:

- **Per-gen zip:** `gen-{gen}-{date}.zip`, ZIP DEFLATE-9. Contains:
  - `MANIFEST.csv` (v0.2 11-column schema, unchanged)
  - `missing.txt`
  - `{bucket}/electricsheep.{gen}.{id}.flam3` — chunked tree mirroring disk
- **Mega-bundle:** `corpus-all-{date}.tar.xz` (xz preset 6 by default —
  good ratio, reasonable build time). Contains the **entire corpus
  tree**:
  - `{gen}/MANIFEST.csv`
  - `{gen}/missing.txt`
  - `{gen}/{bucket}/electricsheep.{gen}.{id}.flam3`
  - `_index/index.json`, `_index/INDEX.md`
  - `ATTRIBUTION.md`
- **Top-level metadata:** `INDEX.md`, `index.json`, `ATTRIBUTION.md`
  (unchanged from v0.3 placement).

`build_date` resolution: `release-build --date YYYY-MM-DD` overrides;
default is `datetime.now(timezone.utc).strftime("%Y-%m-%d")`. The date
is stamped into the top-level of `index.json` as `_build_date` for
post-hoc verification, and into `MANIFEST.csv` is left unchanged (the
11-column schema is locked).

### 4. Index v0.4 — five new pyr3 GPU-safety fields

Added to every genome record in `index.json` (and surfaced in
`INDEX.md` aggregations):

| Field | Type | Source |
|---|---|---|
| `has_hyper_trig` | bool | true if any xform variation ∈ {tan, sec, csc, cot, tanh, sech, csch, coth} |
| `has_edisc` | bool | true if any xform variation == edisc |
| `max_abs_affine_coef` | float | `max(\|coef\|)` across every xform + finalxform `coefs="a b c d e f"` + every post-affine |
| `xform_count_post_symmetry` | int | xform count after applying `<symmetry kind=…>` expansion |
| `has_density_estimator` | bool | true if `<flame estimator_radius=…>` attr present |

Schema version: `index.json` gains a top-level `_schema_version: 4`
field (was implicit v3). `INDEX.md` notes the schema version in the
header.

Each field maps to a documented pyr3 NaN/render failure mode:

| Pyr3 failure mode | Indexed cause |
|---|---|
| f32 denominator cancellation at ±π/2 (`247/00286`, `247/06204`, `247/24344`) | `has_hyper_trig` |
| `sqrt(xmax−1) + log() + acos()` near unit circle → NaN (`247/01978`) | `has_edisc` |
| f32 exponent loss → all-black GPU render (v0.14 HEART/DIAMOND divergence) | `max_abs_affine_coef > 5` |
| GPU chaos pickTable overflow | `xform_count_post_symmetry > 64` |
| Tone-map divergence (soft signal) | `has_density_estimator` |

## Build seam (v0.4, post-implementation)

```
corpus/{gen}/{bucket}/electricsheep.{gen}.{id}.flam3   ← chunked disk
  │
  ├──→  sheep-fold migrate-chunked        (one-shot, idempotent)
  │
  ├──→  sheep-fold index                  (regen _index/{index.json, INDEX.md})
  │     + 5 pyr3 fields + _schema_version: 4
  │
  └──→  sheep-fold release-build [--date YYYY-MM-DD]
        ├──→ build/release/gen-{gen}-DATE.zip       ◀──┐
        │    (DEFLATE-9, per-gen tree)                 │  overlay invariant:
        │                                              ├─ same disk tree
        └──→ build/release/corpus-all-DATE.tar.xz   ◀──┘  after extract,
             (full tree-of-files, xz)                     either path
```

## Out of scope (BACKLOG, v0.5+)

- Runtime NaN detection (`produces_nan: bool`) via short chaos-game
  probe. Either embed a flam3-style iterator in `electric-sheep-fold`
  (new build dep) or ingest a pyr3-side sidecar
  `gen-XXX-nan.txt` (cross-repo coupling). The v0.4 static fields
  above cover the **causes** of NaN; runtime detection catches
  residual false-negatives.
- Tone-mapping diversity fields (`gamma`, `vibrancy`, estimator_*).
  Speculative — no named consumer asking for these queries yet.
- Provenance field (`version` string). Trivial cost; defer until a
  query needs it.
- Per-gen `.tar.xz` artifacts for big gens — redundant given the
  mega-bundle xz already serves bulk consumers.
- Streaming `.flam3` bytes in `release.py:_gather_gen_data` (peak RAM
  ~1.3 GB during gen-244 release-build). Defer until corpus growth
  makes single-machine release-build painful.
- gen 249+ live extension when ES rolls a new live gen. One-line
  invariant update in `LIVE_GENS` in `layout.py`.
